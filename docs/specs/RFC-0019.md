# RFC-0019: CLI/TUI Event Progress Clarity

**RFC**: RFC-0019
**Title**: CLI/TUI Event Progress Clarity
**Status**: Draft
**Kind**: Architecture Design
**Created**: 2026-03-26
**Dependencies**: RFC-0015 (Event System), RFC-0001 (System Conceptual Design)

## Overview

This RFC defines improvements to CLI and TUI event progress rendering to provide clear, readable, and well-structured output for users. The primary change introduces a two-level tree structure for tool calls, where each tool invocation shows a start event (parent) and a completion event (child), with extensible support for special tool behaviors.

## Problem Statement

Current CLI/TUI event progress output has critical usability issues:

1. **Malformatted Text**: AI responses appear without proper spacing ("I'llsearchforinformation...")
2. **Verbose Output**: Large unformatted text blocks overwhelm the terminal
3. **Flat Hierarchy**: All events use same `[tag]` format regardless of importance
4. **Information Density**: Raw XML tags and unsummarized content visible

## Proposed Solution

### Two-Level Tree Structure for Tool Calls

**Default Pattern** (applies to 90% of tools):
```
⚙ WebSearch("Iran wars history conflicts")
  └ ✓ 10 results in 7.6s
```

**Structure**:
- **Level 1 (Parent)**: Tool start with name and key argument(s)
- **Level 2 (Child)**: Tool complete with summary result
- **Indentation**: Use `  └ ` tree connector for child
- **Icons**: ⚙ (in progress) → ✓ (success) or ✗ (error)

**Start Event Format**:
```
⚙ ToolName(key_argument_value)
```
- Extract most relevant argument (path, query, url, command)
- Use `format_tool_call_args()` logic (already exists in `ux/shared/message_processing.py`)
- Show tool display name (CamelCase)
- Yellow color while running, green when complete

**Complete Event Format**:
```
  └ ✓ summary_result
```
- One-line summary of result
- Extract using `extract_tool_brief()` (already exists in `ux/shared/message_processing.py`)
- Truncate to 60 chars max
- Show duration if available

### Special Tool Behaviors

Some tools need additional progress events:

**Browser Subagent** (multi-step):
```
⚙ Browser("iran wars")
  ├ Step 1: navigate @ wikipedia.org
  ├ Step 2: extract_content
  └ ✓ Completed in 45.2s
```

**Research Tool** (multi-phase):
```
⚙ Research("quantum computing")
  ├ Generated 3 queries
  ├ Gathering from arxiv.org
  └ ✓ Synthesized 15 sources (2.3k chars)
```

**Implementation**:
- Create `ToolBehaviorRegistry` to map tool names to custom render functions
- Default behavior for unregistered tools
- Allow tools to emit intermediate progress events as tree nodes

### Text Formatting Fixes

**Issue**: Text concatenation without spaces in AI responses

**Solution**: Add whitespace normalization in `strip_internal_tags()`
- Location: `ux/shared/message_processing.py`
- Add post-processing to ensure proper spacing
- Validate text rendering with spacing tests

### Plan Display Improvements

**Compact tree structure**:
```
● Plan: Search for Iran wars information (1 step)
  └ ⚙ Step 1: Browse web for "Iran wars"
```

**Multi-step Plan**:
```
● Plan: Analyze codebase (3 steps)
  ├ ✓ Step 1: List project structure
  ├ ⚙ Step 2: Read key files
  └ ⏳ Step 3: Generate summary
```

### Protocol Events (Debug Mode Only)

Move protocol events to debug mode:
- Thread lifecycle: debug only
- Context projection: debug only
- Memory storage: debug only
- Checkpoint: debug only

**Normal Mode Focus**:
- User queries
- Tool calls and results
- Plan progress
- Errors

## Architecture Changes

### New Components

1. **ToolCallTracker** (`ux/cli/rendering/tool_call_tracker.py`)
   - Match tool start events with completion events
   - Track tool call state and line indices
   - Support cursor movement for icon updates

2. **ToolBehaviorRegistry** (`ux/cli/rendering/tool_behaviors.py`)
   - Map tool names to custom rendering functions
   - Provide default behavior for unknown tools
   - Extensible for future tools

### Modified Components

1. **CliEventRenderer** (`ux/cli/rendering/cli_event_renderer.py`)
   - Use ToolCallTracker for tree rendering
   - Apply tool behaviors from registry
   - Update event filtering for verbosity

2. **TUI Renderer** (`ux/tui/renderers.py`)
   - Mirror CLI improvements for consistency
   - Use same tool call tracking logic

3. **Message Processing** (`ux/shared/message_processing.py`)
   - Fix `strip_internal_tags()` whitespace normalization
   - Enhance `extract_tool_brief()` for tree rendering

4. **Verbosity Control** (`ux/shared/progress_verbosity.py`)
   - Move protocol events to debug mode
   - Keep tool trees in normal mode

## Success Criteria

1. No malformatted text - all output has proper spacing
2. Two-level tool tree - every tool shows start + complete as parent/child
3. Readable summaries - tool results fit in 1 line, truncated to 60 chars
4. Clear visual hierarchy - icons (⚙ ✓ ✗) indicate status at a glance
5. Terminal-optimized - output respects 80-char width
6. Debug mode preserves detail - all information available when needed
7. Special tools supported - complex tools can add intermediate progress nodes

## Backward Compatibility

- Existing event types and structure unchanged
- No API changes to tool implementations
- Verbosity levels remain (`quiet`, `normal`, `debug`)
- Debug mode shows all original events plus new structure
- TUI mode gets same improvements as CLI
